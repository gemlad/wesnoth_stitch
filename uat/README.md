# UAT — the things that need a person

Everything in this project that a test runner **cannot** settle lands here: questions of the
form "does this actually look right / read right on paper", which need eyes, a printer, and
a verdict. If a decision needs you rather than the compiler, its artefact is in this folder.

Nothing here is a source file. Everything is regenerated by:

```bash
npm run uat              # everything below
npm run uat:chart        # just the export artefacts (fast)
npm run uat:legibility   # just the glyph sheet (slow — re-runs the 7,116-sprite census first)
```

They're committed so you can open them cold without a build. But they are **outputs**: if the
export code changes, regenerate rather than trusting what's checked in.

Point the chart script at any sprite: `npm run uat:chart -- wesnoth-sprites/units/…/x.png`
(default is the dwarvish fighter, `units/dwarves/fighter.png` — 72×72, 31 distinct DMC).

## The artefacts

| File | What it is | What it's for |
| --- | --- | --- |
| `chart-symbol.pdf` | **Black-and-white** chart — cover, floss key, chart page. | **This is [#28].** The glyph is the *only* thing naming a colour, so this is the fair test of the symbol set. |
| `chart-both.pdf` | The working chart — colour underneath, glyph on top. | What you'd really stitch from. Judge whether it's *usable*, not just correct. |
| `preview.png` | The PNG export, 12px per stitch. | The quick-look export. Mainly: does the fabric colour look right behind the pattern? |
| `glyph-legibility-test.pdf` | Seven-page glyph drill — 100mm calibration ruler, the whole set at four cell sizes, the marginal pairs side-by-side *and* separated, a blind identification task with a key. | The systematic version of the same question, from [#20]. |

**The two PDFs are both about glyph legibility, and they disagree in an important way.**
`chart-symbol.pdf` comes from the *real* export path — bundled DejaVu, real A4 geometry. The
legibility sheet predates that export and renders through Chromium's font stack instead. So
**the chart is the authority; the sheet is the drill.** The sheet still earns its place — it's
the only thing that isolates pairs, brackets four sizes and offers a blind test — but where
they disagree, believe the chart.

## How to take the test

**Print at 100% / Actual Size.** Not "fit to page". Not "shrink to printable area". The entire
argument is about *physical millimetres*, and a print dialog that helpfully rescales makes
every verdict below worthless. The legibility sheet's first page carries a **100mm ruler** —
print it, measure it, and if it isn't 100mm, stop and fix the print settings.

The scale that matters is **2.361 mm per cell / 4.82 pt glyphs**: a 72-cell Wesnoth sprite
across A4's printable width. That's where "legible at ~5pt" comes from, and it's what the
charts here are laid out at.

### What to judge

1. **The marginal pairs — `C`/`G`, `E`/`F`, `P`/`R`.** These are the unvalidated survivors of
   the symbol set. In a cell, at speed, without squinting: can you tell them apart?
   **If any pair fails, the colour cap drops.** The glyph count and `MAX_COLOUR_COUNT` are
   *the same number* (37) — a chart can't show more colours than it has symbols to name. So
   losing a glyph costs a colour. Record the verdict on [#28].
2. **Anything else that muddies.** The set was checked at 9px on a screen, never on paper.
3. **Does the chart read as a picture, or as a smudge?** The four solid glyphs (● ■ ▲ ◆) land
   on the four *largest* colour areas, so the densest regions go darkest. This is a known
   problem and it belongs to **[#30]**, not #28 — but this is where you'd see it.

## Where the verdicts go

- Glyph pairs failing in print → comment on **[#28]**, and update §5.3 in `design.md`.
- The chart-reads-as-a-smudge problem, or anything about *which* glyph goes to which colour
  → **[#30]**, the symbol-set spike.
- Anything still undecided → `docs/decisions-pending.md`, which is the queue of calls
  waiting on you.
