# UAT — the things that need a person

Everything in this project that a test runner **cannot** settle lands here: questions of the
form "does this actually look right / read right on paper", which need eyes, a printer, and
a verdict. If a decision needs you rather than the compiler, its artefact is in this folder.

Nothing here is a source file. Everything is regenerated by:

```bash
npm run uat              # everything below
npm run uat:chart        # just the export artefacts
npm run uat:legibility   # just the glyph sheet
npm run uat:legibility -- --letter        # …on US Letter paper
npm run uat:legibility -- --scale=1.0638  # …pre-compensated for a printer that shrinks
```

They're committed so you can open them cold without a build. But they are **outputs**: if the
export code changes, regenerate rather than trusting what's checked in.

Point the chart script at any sprite: `npm run uat:chart -- wesnoth-sprites/units/…/x.png`
(default is the dwarvish fighter, `units/dwarves/fighter.png` — 72×72, 31 distinct DMC).

## The artefacts

| File | What it is | What it's for |
| --- | --- | --- |
| `chart-symbol.pdf` | **Black-and-white** chart — cover, floss key, chart page. | **This is [#28].** The glyph is the *only* thing naming a colour, so this is the fair test of the symbol set. |
| `chart-both.pdf` | The working chart — colour underneath, glyph on top. | What you'd really stitch from. Judge whether it's *usable*, not just correct. |
| `preview.png` | The PNG export, 12px per stitch. | The quick-look export. Mainly: does the fabric colour look right behind the pattern? |
| `glyph-legibility-test.pdf` | Seven-page glyph drill — calibration rulers, all 47 glyphs at four cell sizes, the suspect pairs side-by-side *and* separated, a blind identification task with a key, and two real charts. | The systematic version of the same question. |

**Both PDFs now come from the same place.** The legibility sheet used to render through
Chromium's font stack, which made it a drill rather than an authority. It is now drawn with
`pdf-lib` in exact PDF points, embedding the same DejaVu Sans the export embeds, centring
glyphs with the same maths — so a cell on the sheet and a cell on the chart are the same
object. They can no longer disagree.

## How to take the test

> **Taken and passed (2026-07-23).** #28 is resolved: every letter pair passed, `◆` and `▦`
> were cut (kept `♦` in `◆`'s place), and the cap settled at **47**. The steps below are kept
> so the test can be re-run if the set changes (e.g. under #57). See
> [`../docs/decisions-agreed-2026-07-23.md`](../docs/decisions-agreed-2026-07-23.md) §2.

**Print at 100% / Actual Size.** Not "fit to page", not "shrink oversized pages". The entire
argument is about *physical millimetres*. Page 1 carries a **100 mm bar** (and a 50 mm
cross-check) — print it and measure it before reading anything else.

### If the bar doesn't measure 100 mm

The PDF is provably right: the bar is drawn at exactly 283.4646 pt = 100.0000 mm on an exactly
210×297 mm page. So a short bar means something between the PDF and the paper rescaled it.
Two fixes, in order:

1. **Paper-size mismatch — by far the most common.** Fitting an A4 page onto US Letter shrinks
   everything to 93.9%, landing the bar at ~94 mm. If your printer is loaded with or set to
   Letter, regenerate at that size:
   ```bash
   npm run uat:legibility -- --letter
   ```
2. **Anything else (a driver fitting to the printable area, etc.) — pre-compensate.** Divide
   100 by what you measured and pass it back; the content is drawn that much larger so it
   *prints* at true size. Measured 94 mm:
   ```bash
   npm run uat:legibility -- --scale=1.0638
   ```
   Print again and re-measure — the bar should now be 100 mm. Then take the test.

The scale that matters is **2.361 mm per cell / ~4.8 pt glyphs**: a 72-cell Wesnoth sprite
across the page's printable width. That's where "legible at ~5 pt" comes from, and it's what
the real charts on pages 5–6 are laid out at.

### What to judge

1. **The suspect pairs on page 3.** Three were the original unvalidated survivors — `C`/`G`,
   `E`/`F`, `P`/`R` (all passed). The rest were the glyphs the set gained in the #30/D3
   widening: the 2026-07-23 verdict cut `◆` (kept `♦`) and `▦` (against `■`/`□`), while `♠`
   passed against `▲`, the digits passed, and the print marks passed. Tick "same" only if you
   can't tell them apart **without** comparing side by side.
   **Every glyph that fails is removed, and the colour cap drops by one with it** — the glyph
   count and `MAX_COLOUR_COUNT` are the same number (currently 47). Record the verdict on [#28].
2. **The blind test on page 4.** Errors here matter more than the drill — it's the task a
   stitcher actually performs.
3. **The real charts on pages 5–6.** These now use the **interleaved** assignment rule
   ([#30]/D1, decided). The old complaint — the four solid glyphs landing on the four largest
   areas and the field going black — should be gone; the dominant colour keeps a bold anchor
   while the next-largest areas drop to faint glyphs. If it still reads as a smudge, that's a
   #30 finding, not a #28 one.

## Where the verdicts go

- Glyph pairs failing in print → comment on **[#28]**, and update §5.3 in `design.md`.
- The chart-reads-as-a-smudge problem, or anything about *which* glyph goes to which colour
  → **[#30]**, the symbol-set spike.
- Anything still undecided → a dated `docs/decisions-*.md`. The last round is recorded in
  `docs/decisions-agreed-2026-07-23.md`; all of its calls are now made.
