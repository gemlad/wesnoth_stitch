✗ Milestone 2 — conversion pipeline

Getting the colour right before getting it small

The prototype flattened the image and then snapped to floss, which loses fidelity twice. This pipeline inverts that: map every pixel to real DMC floss first, then reduce by merging the perceptually closest threads. Every colour you see is a real skein, and the slider only ever merges — it never re-clusters, which is why dragging it stays stable.

✗ What the measurements settled

Four numbers that changed the design

The house rule this sprint was measure, don’t assert. Each of these overturned something we believed at the start.

37colour cap
The cap is the number of legible glyphs — not a taste call. A chart can’t show more colours than it has symbols to name them with. We’d guessed 40; the glyphs decided otherwise.
1.14 ΔEcost of the cap
Squeezing the richest sprites down to 37 colours moves the average pixel about half a just-noticeable difference. Colour fidelity was never the binding constraint — glyph legibility is, and it’s the only thing holding the number down.
12.57%of every stitched cell
The half-transparent pixels aren’t anti-aliasing fringe — they’re Wesnoth’s drop shadow, under 90% of sprites. Taken at face value they charted as solid black. Composited over the fabric, they become the grey they look like. This one was visible to the naked eye and nobody had checked.
93.0%of sprites fit
Most sprites fit under the cap with no reduction at all (median: 24 floss colours). But 497 sprites — 1 in 14 — exceed it and must reduce. Reduction isn’t a rare safety net for outliers, as we’d framed it. It runs routinely.

✗ Milestone 3 — export

A chart that knows how big it is

The prototype laid out chart pages as “55 stitches to a page” — a chart with no physical size at all. That matters more than it sounds: the whole legibility argument is stated in millimetres, so a chart that doesn’t know its own size is a chart whose legibility can’t be tested. Every cell is now laid out in real millimetres, and the glyphs are embedded vector text in a bundled font — so a chart prints the same on any machine.

2.361 mmper stitch cell
Not a magic number — a consequence. A4, minus 20 mm margins, divided by a 72-cell Wesnoth sprite. It puts each glyph at 4.82 pt, which is where “legible at ~5 pt” comes from, and it means a standard sprite is exactly one chart page.

✗ Honest notes

Four things that nearly went wrong

Worth recording, because in each case the thing that caught it is the thing worth keeping.

A handover doc that would have undone the symbol set

A note proposed replacing the 37-glyph set with a 52-glyph one. It re-introduced every collision the merged work had already found by rendering real charts — rotated triangles, a star that blurs into a diamond, digits that collide with letters — and would have raised the colour cap past what’s legible.

Caught by: reading the merged code before trusting the document.

The floss key didn’t fit on one page

We’d asserted a full 37-colour key was a single sheet. It was 34 rows. You’d have been turning a page mid-key while threading a needle, to save a millimetre of leading.

Caught by: a test that checked the claim instead of repeating it.

The chart font resolved to nothing outside the app

The way the font was loaded worked when built, but silently produced a broken path anywhere else — meaning the export could be built but never tested. Being testable outside the app is the entire reason we picked this PDF approach, so it was restructured.

Caught by: writing a test for the assembled document.

The print test was hidden in a folder git ignores

The one artefact #28 has always needed — the glyph test sheet — was being written into a build folder that’s wiped on every clean and never committed. It probably explains why that issue has sat open. It now lives in uat/, tracked.

Caught by: asking where the artefacts actually go.

✗ What we need from you

The build isn’t blocked. These are.

Everything you need is in uat/ — regenerate it any time with npm run uat, and uat/README.md tells you what to look at. The running list lives in docs/decisions-pending.md.

  1. Print the chart and judge three letter pairs.

    C/G, E/F, P/R are the unvalidated survivors of the symbol set. Print uat/chart-symbol.pdf at 100% / Actual Size — anything else rescales the page and voids the test. If a pair fails, the colour cap drops by one for each glyph we lose: the glyph count and the colour cap are the same number. (#28)

  2. Tell us what actually failed UAT on the symbols.

    You’ve said the set isn’t right, and the spike is open and waiting on a direction rather than an opinion. The likeliest culprit is already identified: the boldest glyphs are handed to the largest colour areas, so the densest parts of a chart go darkest and it reads as a smudge. The fix is a design choice — possibly the exact opposite rule. (#30)

  3. Is the default colour count what you’d actually stitch?

    Today a pattern defaults to as many colours as the sprite really uses. You’d asked whether a small fixed palette would suit you better. It shipped one way and was never actually decided — and a 30-colour project is a very different evening from a 12-colour one.

  4. When do you want something installable?

    This doesn’t change what gets built, only what comes next: whether packaging gets pulled forward, or whether dev builds are fine for now.
